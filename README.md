# Pure Math Spot Forex Scalping Engine — نسخة المحاكاة (Paper Trading)

## ⚠️ اقرأ هذا أولاً

هذا المشروع في **وضع محاكاة (Paper Trading) فقط**. لا يوجد اتصال بمال حقيقي
ولا بأي وسيط فعلي. كل الأسعار المستخدمة **اصطناعية (Random Walk)** وليست
بيانات سوق حقيقية. النتائج هنا تختبر فقط **صحة منطق الكود رياضيًا** —
وليست وعدًا أو دليلاً على ربح حقيقي متوقع في السوق الفعلي.

---

## ملخص رحلة القرارات (لماذا الإعدادات الحالية هكذا)

| القرار | السبب |
|---|---|
| لا يعمل عبر MT5 | MT5 يتطلب Terminal محلي، لا يعمل على Railway (Linux سحابي) |
| الاتصال المستقبلي عبر REST/WebSocket | متوافق مع النشر السحابي (اقتُرح cTrader Open API) |
| SL=4.0 نقاط / TP=6.0 نقاط | نتيجة بحث منهجي (`param_sweep.py`) — الإعدادات الضيقة الأصلية (SL=1.5/TP=1.2) كانت تخسر دائمًا لأن العمولة ($7/لوت) تلتهم أي هامش ربح صغير |
| عتبة دخول = 2.5 انحراف معياري | إشارات أقل لكن أعلى جودة إحصائية (نتيجة نفس البحث) |
| رافعة افتراضية = 20x | نقطة التوازن بين العائد والمخاطرة (`leverage_sweep.py`) — رافعة أعلى من هذا لم تزد الفرصة فعليًا لأن حجم الصفقة مبني على % مخاطرة من رأس المال، وفقط زادت التذبذب |
| إغلاق إجباري يومي (لم يعد "شرعيًا") | **Gap Risk Guard**: مع رافعة 20x+، فجوة سعرية بسيطة بين إغلاق وافتتاح السوق قد تُصفّي الحساب لأن وقف الخسارة لا يعمل والسوق مغلق |
| حد خسارة يومي 10% | يوقف البوت تلقائيًا في يوم سيئ بدل ترك الخسائر تتراكم بلا حدود |

## هيكل الملفات

```
config.py            # كل الإعدادات القابلة للتعديل في مكان واحد
time_lock.py          # TimeLockManager — حماية فجوة السعر (Gap Risk Guard)
market_scanner.py      # MarketScanner — منطق الدخول الإحصائي (ATR + STD)
paper_broker.py        # PaperBroker — سمسار وهمي لإدارة الصفقات ورأس المال
price_feed.py           # SimulatedPriceFeed — مولّد أسعار اصطناعي للاختبار فقط
execution_engine.py      # ExecutionEngine — الحلقة الرئيسية (asyncio)
main.py                  # نقطة التشغيل + تقرير الأداء
param_sweep.py            # بحث منهجي عن أفضل SL/TP/عتبة دخول (Expectancy)
leverage_sweep.py          # مقارنة مستويات الرافعة على الاستراتيجية المُحسّنة
```

## طريقة التشغيل

```bash
python3 main.py              # محاكاة كاملة + تقرير أداء
python3 param_sweep.py        # إعادة البحث عن أفضل SL/TP (يمكن تعديل الشبكة داخل الملف)
python3 leverage_sweep.py      # إعادة مقارنة مستويات الرافعة
```

لا توجد اعتماديات خارجية — يعمل بـ Python 3.10+ القياسي فقط.

## أهم قيود يجب معرفتها

1. **البيانات اصطناعية بالكامل.** مولّد الأسعار في `price_feed.py` هو Random
   Walk بسيط + حقن "قفزات" عشوائية صغيرة بنسبة 3% من الوقت. الاستراتيجية
   الحالية (SL/TP واسع) تستفيد جزئيًا من نمط هذه القفزات تحديدًا — وهذا
   **قد لا ينطبق على حركة الأسعار الحقيقية**، التي لها بنية مختلفة تمامًا
   (سيولة متغيرة، أخبار، سلوك مؤسسات، إلخ).

2. **لا يوجد اختبار Backtesting على بيانات تاريخية حقيقية بعد.** الخطوة
   التالية الضرورية قبل أي مال حقيقي: تحميل بيانات تاريخية فعلية
   (Tick Data أو 1-minute OHLC) من مصدر موثوق واختبار نفس المنطق عليها.

3. **عينة الاختبار صغيرة إحصائيًا.** نتائج `leverage_sweep.py` مبنية على 3
   بذور عشوائية فقط × 10 أيام محاكاة. هذا لا يكفي لأي استنتاج إحصائي
   موثوق — فقط يعطي اتجاهًا عامًا لمنطق الكود.

4. **لا حماية كاملة من انزلاق السعر (Slippage) الحقيقي.** في السوق
   الفعلي، السعر الذي تُنفَّذ عنده الصفقة قد يختلف عن السعر المطلوب،
   خصوصًا في اللحظات المتقلبة — وهذا غير مُحاكى هنا.

5. **معاملات mean_reversion الحالية (SL=4/TP=6) تُخرج المشروع فعليًا من
   نطاق "السكالبينج فائق السرعة"** إلى نطاق أقرب لـ "التداول قصير المدى"
   (short-term / mini-swing). هذا تغيير هيكلي حقيقي فرضته الرياضيات
   (تكلفة العمولة)، وليس مجرد تفضيل.

## خطوات مقترحة قبل أي مال حقيقي

1. ✅ محاكاة على بيانات اصطناعية (تم) — تختبر فقط صحة الكود
2. ⬜ Backtesting على بيانات تاريخية حقيقية (سنة أو أكثر) لنفس الأزواج
3. ⬜ Forward-testing (Demo حي) على حساب تجريبي حقيقي من وسيط فعلي لمدة
   شهر على الأقل، لمراقبة السلوك في ظروف سوق حقيقية (سبريد متغير، أخبار،
   سيولة فعلية)
4. ⬜ إذا نجحت الخطوتان أعلاه: البدء برأس مال حقيقي صغير يمكن تحمل خسارته
   بالكامل، مع الإبقاء على حد الخسارة اليومي والرافعة المعتدلة

## الخطوة التالية للنشر السحابي (Railway)

الكود الحالي جاهز بنيويًا للربط بمصدر بيانات حقيقي عبر REST/WebSocket
(بدل `SimulatedPriceFeed`). عند الاستعداد لهذه الخطوة، المطلوب فقط:

- استبدال `price_feed.py` بوحدة تتصل بـ API الوسيط الفعلي (مثل cTrader
  Open API عبر WebSocket)
- بقية الملفات (`market_scanner.py`, `paper_broker.py` → يصبح `live_broker.py`,
  `time_lock.py`, `execution_engine.py`) تبقى كما هي بنيويًا

هذا يتيح النشر كخدمة Python طويلة الأمد (Long-running worker) على Railway
بدون أي Terminal محلي.

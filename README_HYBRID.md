# نظام التداول الهجين — OpenApiPy + cTraderFixPy على Railway

## ⚠️ اقرأ هذا قبل أي شيء آخر

هذا **بويلربليت (Boilerplate)** يوضح معمارية صحيحة تقنيًا لدمج الحزمتين،
وليس نظامًا جاهزًا للتداول الحقيقي فورًا. تم التحقق من كل استدعاء API
هنا فعليًا مقابل الحزم المثبتة (وليس من الذاكرة)، لكن **لم يُختبر ضد
اتصال حقيقي بسيرفرات cTrader** لأن بيئة التطوير هنا لا تملك وصولاً
شبكيًا لتلك السيرفرات. اختبره أولاً على حساب **Demo** حصرًا.

---

## 🔑 الحقيقتان التقنيتان اللتان غيّرتا التصميم الأصلي

### 1. FIX API لا يدعم SL/TP إطلاقًا
تحققنا من الكود المصدري لـ `cTraderFixPy` نفسه: رسالة `NewOrderSingle`
لا تحتوي أي حقل لوقف الخسارة أو جني الربح. هذا مؤكد أيضًا من منتدى
cTrader الرسمي. لذلك:
- **الدخول/الخروج** → عبر FIX (`fix_executor.py`) — سريع
- **تثبيت SL/TP** → عبر Open API حصرًا (`ProtoOAAmendPositionSLTPReq`
  في `openapi_streamer.py`) — يُرسل فور تأكيد الفتح عبر FIX

### 2. لا "Multi-threading" — بل Reactor واحد مشترك عبر asyncioreactor
كلا الحزمتين تستوردان `twisted.internet.reactor` عند التحميل (تحققنا
من ذلك بقراءة الكود المصدري المُثبَّت فعليًا). لذلك يجب تثبيت
`asyncioreactor` في **أول سطر تنفيذي** في `main.py` قبل أي استيراد آخر،
وإلا سيُثبَّت الـ reactor الافتراضي تلقائيًا ولن يمكن تغييره لاحقًا.
هذا ما يسمح للحزمتين (وأي كود asyncio مستقبلي) بمشاركة حلقة أحداث واحدة
بأمان تام، بدل المخاطرة بـ Threading حقيقي مع Twisted (غير آمن افتراضيًا).

## هيكل الملفات

| الملف | المسؤولية |
|---|---|
| `config.py` | كل الإعدادات من متغيرات البيئة فقط — لا أسرار في الكود |
| `openapi_streamer.py` | مصادقة، بث أسعار، تعديل SL/TP، معلومات الحساب |
| `fix_executor.py` | تنفيذ أوامر السوق (فتح/إغلاق) بأقصى سرعة |
| `market_scanner.py` | منطق الإشارة الإحصائي البحت (Mean Reversion) |
| `risk_manager.py` | حارس التراجع اليومي 4% + حجم الصفقة القائم على % مخاطرة |
| `telegram_notifier.py` | إشعار كل صفقة + عداد ربح شهري تراكمي (يُصفَّر تلقائيًا) |
| `main.py` | يربط كل شيء على reactor واحد ويشغّل النظام |

## ✅ ما تم التحقق منه فعليًا (وليس افتراضًا)

- تثبيت `ctrader-open-api==0.9.1` و`ctrader-fix==0.1.1` و`Twisted==21.7.0` فعليًا
- فحص `inspect.signature` لكل من `Client.__init__` في الحزمتين
- فحص حقول رسائل Protobuf الفعلية (`ProtoOAApplicationAuthReq`,
  `ProtoOAAccountAuthReq`, `ProtoOASubscribeSpotsReq`, `ProtoOASpotEvent`,
  `ProtoOAAmendPositionSLTPReq`, `ProtoOATraderRes`... إلخ) من الحزمة المثبتة
- فحص الكود المصدري الفعلي لـ `NewOrderSingle._getBody()` في `cTraderFixPy`
  — واكتشفنا أن حقل `TimeInForce` **غير مُضمَّن إطلاقًا** في هذا الإصدار
  رغم وجوده في أمثلة C# الرسمية؛ الكود هنا صُحِّح ليعكس هذا
- بناء كل الكائنات الرئيسية (`OpenApiStreamer`, `FixExecutor`,
  `RiskManager`, `MarketScanner`, `TelegramNotifier`) فعليًا في بيئة
  بايثون حقيقية بدون أخطاء
- اكتشفنا أثناء الاختبار أن `pyopenssl` و`service_identity` مطلوبتان
  فعليًا وإلا يفشل الاتصال بـ `NameError: CertificateOptions` بصمت —
  أُضيفتا لـ `requirements.txt`

## ❌ ما لم يُختبر (لأن بيئة التطوير لا تصل لسيرفرات cTrader)

- الاتصال الفعلي وتسجيل الدخول (Logon) على أي من الحزمتين
- استقبال بث أسعار حقيقي أو تنفيذ أمر حقيقي
- صحة قيم `symbolId` الفعلية لحسابك (تُجلب تلقائيًا من
  `ProtoOASymbolsListRes` عند أول تشغيل — لكن لم نتحقق منها ضد سيرفر حقيقي)
- عمولة دقيقة لكل صفقة (FIX لا يرسلها؛ تحتاج `ProtoOADealListReq` من
  Open API لجلبها بدقة — **علّمناها TODO في الكود ولم ندّعِ خلاف ذلك**)

## خطوات التشغيل

### 1. محليًا (اختبار أولي فقط، لا تُشغّله بحساب Live)
```bash
pip install -r requirements.txt
cp .env.example .env   # املأه ببيانات حساب Demo
python main.py
```

### 2. على Railway
1. ادفع كل هذه الملفات لمستودعك (Forex-Bot)
2. Railway سيكتشف `Dockerfile` تلقائيًا ويبنيه (أو استخدم `Procfile` إن فضّلت Nixpacks)
3. أضف كل المتغيرات من `.env.example` في تبويب **Variables**
4. أنشئ **Volume** واربطه بمسار مثل `/data`، واجعل
   `MONTHLY_COUNTER_PATH=/data/monthly_counter.json` حتى لا يُصفَّر العداد
   عند كل نشر جديد
5. اترك `DRY_RUN=true` لأول تشغيل — راقب سجلات Railway والقنوات على
   تيليجرام قبل التفكير بتفعيل `DRY_RUN=false`

## ⚠️ تذكير أخير (متسق مع كل نقاشاتنا السابقة)

هذا الكود **بنية تحتية صحيحة تقنيًا**، لكنه **لا يحتوي أي دليل على
حافة ربحية حقيقية في السوق**. استراتيجية `market_scanner.py` هي نفسها
التي اختبرناها سابقًا على بيانات اصطناعية فقط. قبل `DRY_RUN=false`
على حساب حقيقي:

1. اختبره أسابيع على حساب **Demo** حقيقي (ليس محاكاة اصطناعية) وراقب
   النتائج الفعلية على بيانات سوق حقيقية
2. تأكد أن `symbolId` المكتشفة تلقائيًا صحيحة لحسابك (راجع اللوج عند أول تشغيل)
3. اطلب من دعم وسيطك تأكيد بيانات FIX (Host/Port/SenderCompID) بدقة
4. ابدأ برأس مال يمكنك تحمل خسارته بالكامل
